# Git and Version Control Systems Guide

## What is VCS?
A **Version Control System (VCS)** is a tool that helps track changes to files over time. It allows multiple developers to collaborate, revert to previous versions, and manage project history efficiently.

### Why VCS?
- Keeps a history of changes
- Enables collaboration across teams
- Prevents accidental data loss
- Simplifies branching and merging workflows

---

## Centralized vs Distributed
- **Centralized VCS (CVCS):**
  - Single central server (e.g., Subversion).
  - Clients check out and commit changes directly to the server.
  - Risk: if the server goes down, collaboration halts.

- **Distributed VCS (DVCS):**
  - Each developer has a full copy of the repository (e.g., Git).
  - Work can continue offline.
  - More resilient and flexible.

---

## Git Mental Model
- **Commit:** A snapshot of your project at a point in time.
- **Branch:** A movable pointer to a commit, used for parallel development.
- **HEAD:** A pointer to the current branch reference.
- **Working Tree vs Staging Area:**
  - *Working Tree:* Your actual files on disk.
  - *Staging Area:* A place to prepare changes before committing.

## Basic Git Commands
```bash
# Initialize a new repository
git init

# Clone an existing repository
git clone https://github.com/user/project.git

# Check status of changes
git status

# Stage files
git add file.txt

# Commit staged changes
git commit -m "Add new feature"

# View commit history
git log

# Compare changes
git diff

# Show details of a commit
git show <commit-hash>
