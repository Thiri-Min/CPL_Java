# Self-Study Guide: Software Engineering Practices

This guide helps you learn modern software engineering practices step by step.  
Each section includes **study instructions**, **websites**, **YouTube videos**, **case studies**, and **notes**.

---

## 1. Git Flow vs Trunk-based Development
- **Git Flow**: Uses multiple long-lived branches (main, develop, feature, release, hotfix). Good for complex release cycles.
- **Trunk-based**: Developers commit directly to main (trunk). Encourages continuous integration and faster delivery.
- **Trade-off**: Git Flow = structured but slower; Trunk-based = faster but requires discipline.

---

## 2. Pull Request / Merge Request
- **Pull Request (GitHub)** and **Merge Request (GitLab)** are proposals to merge code into a branch.
- They allow discussion, review, and automated checks before merging.
- Essential for collaboration and maintaining code quality.

---

## 3. Code Review Basics

- Purpose: Catch bugs, improve readability, share knowledge.
- Best practices: Be respectful, focus on code not person, suggest improvements.
- Reviews improve maintainability and team learning.

---

## 4. Revert vs Reset

- **Revert**: Creates a new commit that undoes changes. Safe for shared repos.
- **Reset**: Moves branch pointer to a previous commit. Can rewrite history—dangerous in shared repos.
- Use revert for collaboration, reset for local cleanup.

---

## 5. Agile Mindset

- Focus on adaptability, collaboration, and delivering value.
- Embrace change instead of resisting it.
- Continuous improvement is key.

---

## 6. Scrum Roles (PO, SM, Dev)

- **Product Owner (PO)**: Defines product vision, manages backlog.
- **Scrum Master (SM)**: Facilitates Scrum, removes impediments.
- **Developers**: Build product increments collaboratively.

---

## 7. Scrum Artifacts (Backlog, Sprint)

- **Product Backlog**: Ordered list of everything needed in the product.
- **Sprint Backlog**: Items selected for the sprint.
- **Increment**: The working product delivered at sprint end.

---

## 8. Scrum Events (Planning, Review, Retro)

- **Sprint Planning**: Decide what to build.
- **Sprint Review**: Show completed work to stakeholders.
- **Retrospective**: Reflect and improve team process.

---

## 9. User Story Format

- Format: *As a [role], I want [feature], so that [benefit]*.
- Keeps focus on user needs and business value.
- Helps prioritize features.

---

## 10. Definition of Ready / Done

- **Definition of Ready (DoR)**: Criteria before work starts (clear, estimated, testable).
- **Definition of Done (DoD)**: Criteria before work is complete (tested, documented, integrated).
- Ensures clarity and quality.

---

## 11. Branch Naming from Ticket

- Branch names should reference ticket IDs for traceability.
- Example: `feature/PROJ-123-add-login`.
- Helps link code changes to tasks/issues.

---

## 12. Commit & PR Linking

- Commits should reference issue IDs (e.g., `fix: PROJ-123 login bug`).
- PRs can be linked to issues for automatic closure.
- Improves transparency and tracking.
